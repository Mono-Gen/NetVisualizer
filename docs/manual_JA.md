# NetVisualizer 操作マニュアル

**対象読者**: ネットワーク初学者  
**バージョン**: 1.0.0

---

## はじめに

NetVisualizer（ネットビジュアライザー）は、TCP/IPネットワークの仕組みを目で見て、手で動かしながら学べる教育用シミュレータです。

このツールを使うと、以下のことができます。

- PC やサーバー、スイッチ、ルーターなどをキャンバス上に配置して、仮想ネットワークを作る
- パケット（データのかたまり）が宛先に届くまでの経路をリアルタイムで可視化する
- ICMP（Ping）、TCP、UDP、DHCP、DNS、IGMP（マルチキャスト）など、主要なプロトコルの動作を体験する
- 設定ミス（サブネットマスク間違い、ゲートウェイ未設定など）がどのような通信障害を引き起こすかを安全に学ぶ

> **ポイント**: 実際のネットワーク機器やルール（RFC）に忠実な動作をシミュレートしています。ここで学んだことは、実際のネットワーク構築に直接役立ちます。

## 第2章　ネットワークを作る（デザインモード）

### 2.1 ノードの種類

| アイコン | 種別 | 役割 |
|----------|------|------|
| 🖥 PC | エンドホスト | ネットワーク通信を発生させる端末 |
| 🗄 Server | サーバー | Webサービス・DNS・DHCP等を提供 |
| 🔀 Switch | L2スイッチ | MACアドレスを学習して転送するL2機器 |
| 🔁 Router | ルーター | 異なるネットワークをつなぐL3機器 |
| 📡 Hub | ハブ | 全ポートにパケットをコピーする旧式機器 |

### 2.2 ノードを配置する

1. デザインモードで、キャンバスの空いている場所を**右クリック**します。
2. メニューからノードの種類を選んで追加します。
3. ノードをドラッグして位置を調整します。

### 2.3 ノードを設定する

ノードをクリックして選択すると、左ペインに設定パネルが開きます。

#### IPアドレスの設定

| 設定項目 | 説明 | 例 |
|----------|------|----|
| IP Address | このノードのIPアドレス | `192.168.1.10` |
| Subnet Mask | サブネットマスク | `255.255.255.0`（/24） |
| Gateway | デフォルトゲートウェイ（別サブネットと通信する場合必須） | `192.168.1.1` |
| DNS | 名前解決に使うDNSサーバーのIPアドレス | `192.168.2.20` |

> **用語解説**: 
> - **IPアドレス**: ネットワーク上の住所。郵便番号のようなもの。
> - **サブネットマスク**: 「同じネットワーク内の仲間かどうか」を判断するための値。
> - **ゲートウェイ**: 別のネットワークへ行くための出口（玄関）にあたるルーターのIPアドレス。

### 2.4 ノード同士をつなぐ（リンクの作成）

1. 接続したいノードの端（ポート）を選択します。
2. 別のノードのポートまでドラッグして放すと、**リンク（LANケーブル）**が作成されます。

> **注意**: スイッチやルーターは複数のポートを持っています。ノードの右クリックメニューから「インターフェースを追加」することでポート数を増やせます。

### 2.5 正しいトポロジーの例

```
[PC 1] ─── [Switch] ─── [Router] ─── [Server]
 192.168.1.10    L2転送  192.168.1.1/192.168.2.1  192.168.2.20
  /24                     GW:192.168.1.1→               /24
```

## 第3章　通信を試す（シミュレーションモード）

`▶ Simulate` ボタンを押してシミュレーションモードに切り替え、通信させたいノードをクリックします。

### 3.1 Ping（ICMP Echo Request/Reply）

**Pingとは**: 「相手に届くか？」を確認する最も基本的な疎通確認ツールです。

#### 操作手順

1. 送信元のノードをクリックして選択します。
2. 左ペインの「COMMAND FOR [ノード名]」欄の下に表示されるIPアドレス入力欄に、**宛先のIPアドレス**（例: `192.168.2.20`）を入力します。
3. `Ping` ボタンを押します。

#### 観察ポイント

- キャンバス上を飛び回るパケット（丸いアニメーション）を追ってみましょう。
- 最初に **ARP Request** が飛ぶことに注目してください（次のホップのMACアドレスを知るため）。
- 正常に届けば `echo-reply` が返ってきます。

#### よくある失敗パターン

| エラーログ | 原因 | 対処 |
|----------|------|------|
| `No Gateway` | ゲートウェイが未設定 | デザインモードでGatewayを設定 |
| `TTL Expired` | ルーターを通りすぎた（ループの可能性） | 経路設定を見直す |
| `Host Unreachable` | ルートが見つからない | ルーターのインターフェース設定を確認 |

### 3.2 UDP送信（L4メッセージング）

**UDPとは**: 送りっぱなしの通信です。確認応答がなく、高速ですが信頼性は低いです。  
動画ストリーミングや音声通話など、少し欠けても問題ない用途に使われます。

#### 操作手順

1. 送信元ノードを選択します。
2. 左ペインの「L4 MESSAGING」欄でポート番号（例: `8080`）とメッセージを入力します。
3. `UDP Send` ボタンを押します。

#### 観察ポイント

- 受信先のノードがそのポートを**受信待機（Listen）していない**場合、`ICMP Port Unreachable`（ポート到達不能）エラーが返ってきます。
- 受信待機しているサーバーを使う場合は、デザインモードで「サービス」タブからポートを開放してください。

> **注意（RFC準拠）**: マルチキャスト（`224.x.x.x`）宛てのUDPパケットは、ポートが閉じていても **ICMPエラーを返しません**。これは、大量のエラーでネットワークが溢れることを防ぐためのルールです（RFC 1122）。

### 3.3 TCP通信（3-way ハンドシェイク）

**TCPとは**: 「接続を確立してからデータを送る」信頼性の高い通信です。  
WebブラウザやメールなどはこのTCPを使っています。

#### 操作手順

1. 送信元ノードを選択します。
2. ポート番号（例: `80`）を入力します。
3. `TCP Connect` ボタンで接続を開始します。

#### 通信の流れ（3-way ハンドシェイク）

```
PC（クライアント）          サーバー
      │                       │
      │── SYN ──────────────>│  「接続したいです」
      │<─ SYN-ACK ──────────│  「了解しました」
      │── ACK ──────────────>│  「接続確立！」
      │                       │
      │== ESTABLISHED (通信開始) ==
      │                       │
      │── TCP Send ──────────>│  データ送信
      │                       │
      │── FIN ──────────────>│  「切断したいです」
      │<─ FIN-ACK ──────────│  「了解しました」
```

## 第4章　ネットワークサービス

### 4.1 DHCP（IPアドレス自動取得）

**DHCPとは**: 「自動でIPアドレスをもらう」仕組みです。  
家のWi-Fiにつないだとき、自動でIPアドレスが振られるのはDHCPのおかげです。

#### 事前準備（デザインモード）

DHCPサーバーを配置し、以下を設定します。
- **IP Pool（配布範囲）**: 例 `192.168.1.100` 〜 `192.168.1.200`
- **Lease Time**: IPを貸し出す期間
- **Gateway/DNS**: クライアントに渡す情報

#### 操作手順（DORAプロセス）

1. IPアドレスが `0.0.0.0` のPCを選択します。
2. `DHCP Renew` ボタンを押します。

#### DORAの流れ

```
PC                               DHCPサーバー
 │── DISCOVER (broadcast) ──────>│  「誰かIPくれますか？」
 │<─ OFFER ───────────────────── │  「192.168.1.100どうぞ」
 │── REQUEST (broadcast) ──────> │  「じゃあそれをもらいます」
 │<─ ACK ──────────────────────  │  「確定しました！」
```

> **ポイント**: DHCP通信はブロードキャスト（全員への送信）です。そのため、ルーターを越えては届きません（TTL=1）。DHCPサーバーはクライアントと同じL2セグメント（同じスイッチ配下）に置く必要があります。

### 4.2 mDNS Query（マルチキャストDNS）

**mDNSとは**: DNSサーバーなしで、ネットワーク上のホスト名を解決する仕組みです。  
Appleの「Bonjour」や、プリンターの自動検出などに使われています。

#### 操作手順

1. 検索したいノードを選択します。
2. 左ペインの「NETWORK SERVICES」から `mDNS Query` を押します。

#### 通信の流れ

```
PC                           グループ 224.0.0.251
 │── mDNS Query (multicast) ───────────────────>│
 │                                               │
 │                          応答するサーバー
 │<─ mDNS Response (unicast) ─────────────────  │
```

> **条件**: 応答するノードは「mDNS RESPONDER」が **ENABLED** に設定されている必要があります。また、そのノードが `224.0.0.251` グループに参加していることが前提です。

## 第5章　マルチキャストと IGMP

### 5.1 マルチキャストとは

**マルチキャスト**は「特定のグループに登録した全員に同時配信する」通信方式です。

| 通信方式 | 宛先 | 例 |
|----------|------|-----|
| ユニキャスト | 1対1 | 電話 |
| ブロードキャスト | 全員 | ラジオ |
| **マルチキャスト** | グループのメンバー全員 | 有料チャンネル |

### 5.2 マルチキャストグループへの参加（Join）

マルチキャストを**受け取る**には、まずそのグループに参加（Join）する必要があります。  
送る側はグループに参加していなくても送信できます（テレビ局は自分でテレビを見なくてもよい）。

#### 操作手順

1. グループに参加させたいノードを選択します。
2. 左ペインの「MULTICAST GROUP」欄にグループアドレス（例: `224.2.2.2`）を入力します。
3. `Join` ボタンを押します。
4. ノードの下に **「JOINED GROUPS」** という黄色いバブルが表示されれば成功です。

> **裏側で何が起きているか**: `Join` を押すと、裏側では **IGMP Membership Report** パケットが自動的にスイッチに送信されます。これによりスイッチが「このポートの先にグループのメンバーがいる」と学習します。

### 5.3 グループから退出（Leave）

1. 同じパネルで `Leave` ボタンを押すと、グループから退出します。
2. スイッチのIGMP Snoopingテーブルからも自動的に削除されます。

### 5.4 IGMP Snoopingの観察

スイッチがマルチキャストパケットを「賢く転送」するために行うのが **IGMP Snooping**（IGMPスヌーピング）です。

#### 確認方法

1. ノード（Join済み）から先のスイッチを確認します。
2. スイッチの下に **「IGMP SNOOPING」** テーブルが表示されます。
3. グループアドレスと、そのグループに参加しているポートが一覧表示されます。

```
┌──────────────────────────────┐
│       IGMP SNOOPING          │
│  224.2.2.2 → eth0, eth2     │
│  224.0.0.251 → eth0, eth3   │
└──────────────────────────────┘
```

> **なぜ重要か**: IGMP Snoopingがなければ、マルチキャストパケットは全ポートにブロードキャストされてしまいます（ネットワークが混雑する）。スヌーピングにより、必要なポートだけに転送することで効率化しています。

## 第6章　学習用テーブルの読み方

通信が発生すると、各ノードの下に「学習した情報」が自動的に表示されます。

### 6.1 ARP TABLE（ARPテーブル）

- **表示ノード**: PC、サーバー、ルーター
- **内容**: IPアドレス ↔ MACアドレスの対応表
- **役割**: 「IPアドレスは知っているが、実際にパケットを届けるにはMACアドレスが必要」な場合に参照される

```
┌────────────────────────┐
│       ARP TABLE        │
│  192.168.1.1 → AA:BB  │  ← ゲートウェイのMACアドレス
│  192.168.1.11 → CC:DD │  ← 同じサブネット内のPCのMACアドレス
└────────────────────────┘
```

> **用語解説（ARP）**: Address Resolution Protocol の略。IPアドレスからMACアドレスを調べるプロトコル。「192.168.1.10 は誰ですか？」とブロードキャストで問い合わせ、本人が「私のMACアドレスは〇〇です」と返答する。

### 6.2 MAC TABLE（MACテーブル）

- **表示ノード**: スイッチ
- **内容**: MACアドレス → ポート番号の対応表
- **役割**: どのポートの先にどのMACアドレスの機器がいるかを記録し、効率よく転送する

```
┌────────────────────────┐
│       MAC TABLE        │
│  AA:BB:CC → eth0       │  ← eth0の先にAA:BB:CC がいる
│  DD:EE:FF → eth1       │
└────────────────────────┘
```

> **なぜ必要か**: スイッチはMACテーブルがないと全ポートにパケットをコピー（フラッド）してしまいます。MACを学習することで、必要なポートだけに転送できるようになります。

### 6.3 DNS CACHE（DNSキャッシュ）

- **表示ノード**: PC、サーバー
- **内容**: ホスト名 → IPアドレスの変換結果のキャッシュ
- **役割**: 一度解決した名前を記憶しておくことで、次回の通信を高速化する

```
┌────────────────────────────────┐
│          DNS CACHE             │
│  server1.local → 192.168.2.20 │
└────────────────────────────────┘
```

### 6.4 JOINED GROUPS（参加中のマルチキャストグループ）

- **表示ノード**: PC、サーバー
- **内容**: 現在参加しているマルチキャストグループのIPアドレス一覧

```
┌──────────────────────┐
│    JOINED GROUPS     │
│    224.0.0.251       │  ← mDNSグループ
│    224.2.2.2         │  ← カスタムグループ
└──────────────────────┘
```

## 第7章　PACKET LOGS（パケットログ）の読み方

画面下部のパケットログには、発生した通信イベントが時系列で表示されます。

### ログの種類と意味

| 色 / 種別 | 意味 |
|-----------|------|
| **青 (RCVD/received)** | パケットが正常に受信された |
| **緑 (sent)** | パケットが送信された |
| **赤 (ERROR)** | エラーが発生した（パケット到達不能、ポート未開放など） |
| **グレー (dropped)** | パケットが意図的に破棄された（マルチキャスト未参加、L2フィルタリング等） |

### ログの読み方

```
[ERROR]  21:24:56
169.254.181.224 → 224.2.2.2
UDP Port 8080 closed. Sending ICMP Port Unreachable.
```

- **送信元IP** → **宛先IP**: 誰から誰へ送ったか
- **説明文**: 何が起きたか

## 第8章　トラブルシューティングガイド

通信が失敗したとき、以下のチェックリストを順番に確認してください。

### ✅ L2（物理・MACアドレス）のチェック

- [ ] リンク（ケーブル）は正しくつながっているか？
- [ ] スイッチのMACテーブルに宛先MACが学習されているか？

### ✅ L3（IPアドレス・ルーティング）のチェック

- [ ] 送信元と宛先のIPアドレスは同じサブネットか？（違う場合はゲートウェイが必要）
- [ ] ゲートウェイのIPアドレスは正しく設定されているか？
- [ ] ゲートウェイのIPアドレスは自分と同じサブネット内にあるか？
- [ ] ルーターの両側にIPアドレスが正しく設定されているか？

### ✅ L4（ポート）のチェック

- [ ] 宛先サーバーで、接続しようとしているポートが開いているか？（TCPポート開放設定）

### ✅ マルチキャスト特有のチェック

- [ ] 受信側のノードは対象グループに参加（Join）しているか？
- [ ] `mDNS RESPONDER` は ENABLED になっているか？

### よくある間違いと正しい設定

#### ❌ 間違い: ゲートウェイがサブネット外
```
PC:      IP=192.168.1.10  Mask=255.255.255.0  GW=10.0.0.1  ← 10.x.xはサブネット外
```

#### ✅ 正しい設定
```
PC:      IP=192.168.1.10  Mask=255.255.255.0  GW=192.168.1.1  ← 同じ /24 内
Router:  IP=192.168.1.1（LAN側）/  10.0.0.1（WAN側）
```

## 付録A：主要プロトコルまとめ

| プロトコル | 階層 | 用途 | RFCの参考番号 |
|-----------|------|------|--------------|
| ARP | L2/L3境界 | IPからMACを解決 | RFC 826 |
| ICMP | L3 | 疎通確認・エラー通知 | RFC 792 |
| IGMP | L3（マルチキャスト制御） | グループ管理 | RFC 1112, RFC 2236 |
| UDP | L4 | 軽量・高速なデータ転送 | RFC 768 |
| TCP | L4 | 信頼性の高いデータ転送 | RFC 793 |
| DHCP | L7（UDP上） | IP自動割り当て | RFC 2131 |
| DNS | L7（UDP/TCP上） | 名前解決 | RFC 1035 |
| mDNS | L7（UDP/マルチキャスト） | ローカル名前解決 | RFC 6762 |

| ノードの選択 | ノードをクリック |
| ノードの移動 | ドラッグ |
